THE 45-SECOND VERSION

What it is, how it works, and why it was hard.

WHAT

Monitoring & control for distributed energy assets

Generators, inverters, and meters at commercial & industrial sites in Nigeria. Telemetry comes in over Modbus; decisions go out through an approval queue.

HOW

An MCP server puts the site in the agent's hands

The MCP server exposes telemetry and control tools to an LLM agent. The agent detects anomalies, plans an intervention, and files it for human sign-off in a React dashboard. Prometheus and Grafana watch the watcher.

WHY IT'S HARD

Crashes meet physical actuators

The hard problem isn't getting an agent to call tools. It's guaranteeing a crashed run resumes instead of re-firing a command at a device controlling real equipment. Run state lives in PostgreSQL, not in memory.

ARCHITECTURE

Drawn as a one-line diagram, because that's how it behaves.

The approval gate is rendered the way electricians render it: a normally-open breaker between the agent and the hardware. Current — a command — only flows when a human closes it.

THE HARD PROBLEM

Restarts must never re-fire hardware commands.

A stateless agent that crashes mid-run and replays its plan is an inconvenience when it's calling a search API. It's an incident when it's talking to a generator. Every PowerOps run is a state machine persisted to PostgreSQL — each transition is recorded, so recovery knows what was proposed, what was approved, and what was already dispatched, and resumes from there instead of starting over.

GUARANTEE

Approval is structural, not cosmetic

The execution path is blocked until a human approves. The agent's tool surface lets it propose — never dispatch.

GUARANTEE

Runs resume, commands don't repeat

Run state is durable in Postgres. A crash mid-run resumes from the last recorded transition — it never replays a dispatched command.

GUARANTEE

The agent is observable

Prometheus counters and Grafana dashboards cover the loop — proposals, approvals, rejections, outcomes — so trust is inspectable, not assumed.

AUTHORSHIP

What I personally built: all of it.

PowerOps is a solo project — every layer below is mine, end to end.

AGENT CORE + MCP

The MCP server and the tool surface the agent works through — telemetry reads, action proposals, run lifecycle. Designed so the most an agent can ever do on its own is ask.

CONTROL PLANE

The Express API and Modbus integration that actually talk to devices in the field — the only component allowed to touch hardware, and only with an approved run behind it.

DATA LAYER

The PostgreSQL schema carrying telemetry, persisted run state, and the approvals ledger — the source of truth that makes crash recovery boring.

OPERATOR DASHBOARD

The React/Vite frontend: the approval queue where humans close the gate, plus per-site views of what the agent is seeing and doing.

OBSERVABILITY

Prometheus metrics and Grafana dashboards across the whole loop, so the agent's behavior is a graph you can look at, not a vibe.

Where it comes from: PowerOps grew out of FloWiser, a commercial-energy venture I co-founded in Nigeria. The problem is real and so is the hardware — which is why the platform is built around approval gates and resumable runs rather than a demo loop.
